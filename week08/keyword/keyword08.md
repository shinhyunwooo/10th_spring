- Spring Security가 무엇인가?
    
    Spring Security는 Spring 기반 애플리케이션에서 **인증(Authentication)** 과 **인가(Authorization)** 를 처리하기 위한 보안 프레임워크이다. 로그인, 권한 관리, 세션 처리, CSRF 방어 등의 기능을 제공하며 웹 애플리케이션의 보안 기능을 쉽게 구현할 수 있다. 또한 Filter 기반 구조를 사용하여 요청이 Controller에 도달하기 전 사용자의 인증 여부와 권한을 검사한다.
    
    **특징**
    
    - 인증 및 인가 기능 제공
    - 세션 관리 지원
    - CSRF, XSS 등 보안 기능 제공
    - JWT, OAuth2 같은 인증 방식과 연동 가능
    
    **장점**
    
    - 강력한 보안 기능 제공
    - Spring 생태계와 높은 호환성
    - 확장성이 뛰어남


- 인증(Authentication)vs 인가(Authorization)
    
    인증(Authentication)은 사용자가 **누구인지 확인하는 과정**이며, 인가(Authorization)는 인증된 사용자가 **어떤 권한을 가지는지 확인하는 과정**이다.
    
    예를 들어 로그인 시 아이디와 비밀번호를 입력하여 신원을 확인하는 과정은 인증에 해당하며, 로그인 이후 관리자 페이지 접근 권한을 검사하는 과정은 인가에 해당한다.
    
    | 구분 | 인증(Authentication) | 인가(Authorization) |
    | --- | --- | --- |
    | 의미 | 사용자 신원 확인 | 권한 확인 |
    | 질문 | "당신은 누구인가?" | "무엇을 할 수 있는가?" |
    | 예시 | 로그인 | 관리자 페이지 접근 |
    
    Spring Security는 일반적으로 **인증 → 인가 순서**로 처리한다. 인증이 완료되어야 권한 검사가 가능하다.


    
    
- Stateful vs Stateless
    
    Stateful은 서버가 클라이언트의 상태 정보를 저장하는 방식이고, Stateless는 서버가 상태를 저장하지 않는 방식이다.
    
    **Stateful**
    
    사용자의 로그인 정보나 세션 정보를 서버가 직접 저장한다. 대표적으로 Session 기반 인증 방식이 사용된다.
    
    장점:
    
    - 사용자 정보 관리가 쉬움
    - 서버에서 직접 상태 관리 가능
    
    단점:
    
    - 서버 메모리 사용량 증가
    - 서버 확장 시 세션 공유 문제 발생
    
    **Stateless**
    
    서버가 사용자 상태를 저장하지 않으며, 요청마다 인증 정보를 함께 전달한다. JWT 방식이 대표적이다.
    
    장점:
    
    - 서버 확장에 유리
    - 메모리 사용량 감소
    
    단점:
    
    - 요청마다 토큰 전송 필요
    - 토큰 탈취 시 위험 존재
    
    최근 REST API 환경에서는 서버 확장성과 유지보수 측면에서 **Stateless + JWT 방식**이 많이 사용된다.
