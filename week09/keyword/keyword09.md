- 세션과 토큰의 차이는?
    
    
    | 항목 | 세션(Session) | 토큰(Token) |
    | --- | --- | --- |
    | 저장 위치 | 서버 | 클라이언트 |
    | 인증 방식 | 서버가 사용자 상태 저장 | 토큰 자체가 인증 정보 포함 |
    | 서버 부담 | 큼 | 적음 |
    | 확장성 | 낮음 | 높음 |
    | 모바일/API | 상대적으로 불편 | 매우 적합 |
    | 대표 사용 | 전통적인 웹 로그인 | JWT 인증 |
    
    ### 세션 방식
    
    사용자가 로그인하면 서버가 세션을 생성하고 세션 ID를 발급한다. 클라이언트는 세션 ID만 저장하며 요청 시 전달한다. 서버는 저장된 세션을 조회해 사용자를 인증한다.
    
    동작:
    
    ```
    로그인
    ↓
    서버 세션 생성
    ↓
    세션ID 발급
    ↓
    클라이언트 저장
    ↓
    요청 시 세션ID 전달
    ```
    
    ### 토큰 방식
    
    로그인 성공 시 서버가 JWT 토큰을 발급한다. 클라이언트는 토큰을 저장하고 요청 시 Header에 포함한다.
    
    ```
    Authorization: Bearer JWT토큰
    ```
    
    동작:
    
    ```
    로그인
    ↓
    JWT 발급
    ↓
    클라이언트 저장
    ↓
    요청 시 토큰 전달
    ↓
    토큰 검증
    ```
    
    ### 핵심 차이
    
    세션은 **서버가 사용자 상태를 기억**하고, 토큰은 **토큰 자체가 인증 정보 역할**을 한다.
    
- 엑세스 토큰과 리프레시 토큰이란?
    
    JWT 인증에서는 보안을 위해 보통 두 종류 토큰을 사용한다.
    
    ## Access Token
    
    실제 API 인증에 사용하는 토큰
    
    특징:
    
    - 유효시간 짧음
    - API 요청마다 사용
    - 탈취 위험 고려
    
    예:
    
    ```
    30분
    1시간
    2시간
    ```
    
    사용:
    
    ```
    Authorization: Bearer AccessToken
    ```
    
    ---
    
    ## Refresh Token
    
    Access Token이 만료되었을 때 새 토큰 발급용
    
    특징:
    
    - 유효시간 김
    - API 호출에는 사용하지 않음
    - 재로그인 감소
    
    예:
    
    ```
    7일
    14일
    30일
    ```
    
    동작:
    
    ```
    로그인
    ↓
    Access + Refresh 발급
    ↓
    Access 만료
    ↓
    Refresh로 새 Access 재발급
    ```
    
    ### 핵심 차이
    
    | 항목 | Access Token | Refresh Token |
    | --- | --- | --- |
    | 목적 | API 인증 | 토큰 재발급 |
    | 사용 빈도 | 많음 | 적음 |
    | 유효시간 | 짧음 | 김 |
    | 노출 위험 | 높음 | 낮음 |
- OAuth 1.0과 OAuth 2.0의 차이는?
    
    OAuth는 다른 서비스 계정을 이용해 로그인하는 방식이다.
    
    예:
    
    - 카카오 로그인
    - 구글 로그인
    - 네이버 로그인
    
    ---
    
    ## OAuth 1.0
    
    특징:
    
    - 요청마다 서명(Signature) 생성
    - 구현 복잡
    - 보안 높음
    - 사용 어려움
    
    방식:
    
    ```
    Consumer Key
    +
    Secret Key
    +
    Signature
    ```
    
    문제:
    
    구현이 매우 복잡하고 개발 난이도가 높았다.
    
    ---
    
    ## OAuth 2.0
    
    특징:
    
    - HTTPS 사용
    - 구현 간단
    - 현재 표준
    - 모바일/API 환경 적합
    
    방식:
    
    ```
    Client ID
    +
    Access Token
    ```
    
    현재:
    
    - Google 로그인
    - 카카오 로그인
    - GitHub 로그인
    
    거의 전부 OAuth 2.0 사용
    
    ---
    
    ### 차이 비교
    
    | 항목 | OAuth 1.0 | OAuth 2.0 |
    | --- | --- | --- |
    | 보안 방식 | Signature | HTTPS |
    | 구현 난이도 | 어려움 | 쉬움 |
    | 모바일 지원 | 불편 | 좋음 |
    | 사용 여부 | 거의 사용 안 함 | 표준 |
