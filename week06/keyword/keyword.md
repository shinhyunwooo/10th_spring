- JPA란?
    
    JPA(Java Persistence API)란 자바 객체와 관계형 데이터베이스 테이블을 매핑해주는 ORM(Object-Relational Mapping) 표준 인터페이스다.
    
    1. JPA는 인터페이스(명세)이고, 실제 구현체는 Hibernate가 가장 널리 쓰인다. 즉 JPA를 쓴다는 건 보통 Hibernate를 쓴다는 뜻이다.
    2. 개발자가 SQL을 직접 작성하지 않아도, 객체를 저장·조회·수정·삭제하면 JPA가 알아서 SQL을 생성하고 실행해준다.
    3. 장점은 생산성 향상, 객체 중심 개발, DB 종류에 독립적이라는 것이다. 단점은 복잡한 쿼리 최적화가 어렵고, 잘못 쓰면 성능 문제(N+1 등)가 생긴다.
- N+1 문제란?
    
    연관 엔티티를 조회할 때, 1번의 쿼리로 N개의 결과를 가져온 뒤 각각에 대해 추가로 N번의 쿼리가 발생하는 성능 문제다.
    
    1. 예를 들어 Member 10명을 조회(1번)한 뒤, 각 멤버의 Order 목록을 가져오기 위해 10번의 쿼리가 추가 실행되는 상황이 N+1이다. 총 11번 실행됨.
    2. 지연 로딩(Lazy Loading) 설정 시 특히 자주 발생한다. 연관 객체에 실제로 접근하는 순간 쿼리가 나가기 때문이다.
    3. 해결 방법으로는 Fetch Join, @EntityGraph, Batch Size 설정 등이 있다.
- 지연로딩과 즉시로딩의 차이는?
    
    연관 엔티티를 언제 DB에서 가져오느냐의 전략 차이다.
    
    1. 즉시 로딩(EAGER): 부모 엔티티를 조회하는 순간, 연관된 엔티티도 함께 JOIN해서 바로 가져온다. 코드가 단순해지지만 불필요한 데이터까지 다 로딩될 수 있다.
    2. 지연 로딩(LAZY): 부모 엔티티만 먼저 가져오고, 연관 엔티티는 실제로 접근하는 시점에 쿼리가 나간다. 성능상 유리하지만 N+1 문제를 주의해야 한다.
    3. JPA 기본 권장 전략은 모두 LAZY로 설정하고, 필요한 경우에만 Fetch Join으로 함께 가져오는 것이다. `@ManyToOne`은 기본이 EAGER라 주의가 필요하다.
- JPQL란?
    
    SQL과 유사하지만, 테이블이 아닌 엔티티 객체를 대상으로 작성하는 JPA 전용 쿼리 언어다.
    
    1. `SELECT m FROM Member m WHERE m.age > 20` 처럼 테이블명 대신 클래스명, 컬럼명 대신 필드명을 사용한다.
    2. JPA가 JPQL을 실제 DB 방언(dialect)에 맞는 SQL로 변환해서 실행하기 때문에, DB 종류가 바뀌어도 JPQL 코드는 그대로 유지된다.
    3. 단점은 문자열로 쿼리를 작성하기 때문에 컴파일 시점에 오류를 잡지 못한다는 것이다. 이를 해결하기 위해 Criteria API나 QueryDSL을 사용한다.
- Fetch Join란?
    
    JPQL에서 연관 엔티티를 한 번의 쿼리로 함께 가져오도록 명시하는 방법으로, N+1 문제의 핵심 해결책이다.
    
    1. 일반 JOIN은 연관 엔티티를 SELECT 대상에 포함시키지 않지만, Fetch Join은 연관 엔티티까지 SELECT해서 영속성 컨텍스트에 올려준다.
    2. 사용법: `SELECT m FROM Member m JOIN FETCH m.orders` — 이렇게 하면 Member와 Order를 한 방에 가져온다.
    3. 단점으로는 페이징(Pagination)과 함께 쓸 때 문제가 생긴다. 컬렉션 Fetch Join + 페이징을 동시에 하면 Hibernate가 메모리에서 전체를 가져와 처리하며 경고를 낸다. 이때는 `@BatchSize`나 `@EntityGraph`를 고려한다.
- @EntityGraph란?
    
    Fetch Join을 어노테이션으로 선언적으로 표현하는 방법으로, Repository 메서드에 붙여서 로딩 전략을 오버라이드한다.
    
    1. 기본 설정은 LAZY지만, 특정 조회 메서드에서만 연관 엔티티를 같이 가져오고 싶을 때 사용한다.
    2. 사용법: `@EntityGraph(attributePaths = {"orders"})` 를 Repository 메서드 위에 붙이면 해당 메서드 호출 시 orders를 함께 로딩한다.
    3. Fetch Join보다 코드가 간결하고 JPQL 없이 쓸 수 있다는 장점이 있다. 단, 복잡한 조건 쿼리에서는 Fetch Join이 더 유연하다.
- commit과 flush 차이점은?
    
    둘 다 DB에 반영하는 것처럼 보이지만, 동작 시점과 범위가 다르다.
    
    |  | **flush** | **commit** |
    | --- | --- | --- |
    | 역할 | 영속성 컨텍스트의 변경사항을 **SQL로 DB에 전송** | 트랜잭션을 **최종 확정**하여 DB에 영구 저장 |
    | 롤백 가능? | 가능 (트랜잭션 유지) | 불가 (트랜잭션 종료) |
    | 영속성 컨텍스트 초기화? | 유지됨 | 기본적으로 초기화 |
    1. **flush**는 영속성 컨텍스트에 쌓인 변경 감지(dirty checking) 결과를 SQL로 DB에 **보내는 것**이다. 하지만 트랜잭션이 아직 살아있으므로 롤백이 가능하다.
    2. **commit**은 트랜잭션을 끝내면서 지금까지의 모든 작업을 **DB에 영구 확정**하는 것이다. commit 직전에 flush가 자동으로 호출된다.
    3. 즉 순서는 `flush(SQL 전송)` → `commit(트랜잭션 확정)`이다. flush 없이 commit만 해도 내부적으로 flush가 먼저 실행된다.
